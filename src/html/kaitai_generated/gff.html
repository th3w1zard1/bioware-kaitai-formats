Gff format specification

Type: Gff

BioWare **GFF** (Generic File Format): hierarchical binary game data (KotOR/TSL and Aurora lineage; GFF4 for DA / Eclipse-class payloads in this `.ksy`). Human-readable tables and tutorials: PyKotor wiki (**Further reading**). Wire `gff_field_type` enum: `formats/Common/bioware_gff_common.ksy`. **Aurora prefix (8 bytes):** `u4be` FourCC + `u4be` version (`AuroraFile::readHeader` — `meta.xref` `xoreos_aurorafile_read_header`). **GFF3:** Twelve LE `u32` counts/offsets as `gff_header_tail` under `gff3_after_aurora`, then lazy arena `instances`. **GFF4:** When version is `V4.0` / `V4.1`, the next field is `platform_id` (`u4be`), not GFF3 `struct_offset` (`gff4_after_aurora`; partial GFF4 graph — `tail` blob still opaque). **GFF3 wire summary:** - Root `file` → `gff_union_file`; arenas addressed via `gff3.header` offsets. - 12-byte struct rows (`struct_entry`), 12-byte field rows (`field_entry`); root struct index **0**; single-field vs multi-field vs lists per wiki *Struct array* / *Field indices* / *List indices*. **Ghidra / VMA:** engine record names and addresses live on the `seq` / `types` nodes they justify, not in this blurb. **Pinned URLs and tool history:** `meta.xref` (alphabetical keys). Coverage matrix: `docs/XOREOS_FORMAT_COVERAGE.md`.

OffsetSizeIDTypeNote
0 ... file GffUnionFile Aurora container: shared **8-byte** prefix (`u4be` magic + `u4be` version tag), then either **GFF3** (`gff3_after_aurora`: 48-byte `gff_header_tail` + arena `instances`) or **GFF4** (`gff4_after_aurora`). Discrimination matches xoreos `loadHeader` order (`gff3file.cpp` vs `gff4file.cpp`); Kaitai uses mutually exclusive `if` on `seq` fields (V4.* vs non-V4) so `gff3` / `gff4` have stable types for downstream `pos:` / `_root.file.gff3.header` paths.

Type: FieldArray

Table of `GFFFieldData` rows (`field_count` × 12 bytes at `field_offset`). Indexed by struct metadata and `field_indices_array`. Cross-check: https://github.com/OpenKotOR/PyKotor/blob/master/Libraries/PyKotor/src/pykotor/resource/formats/gff/io_gff.py#L163-L180 (`_load_fields_batch` reads 12-byte headers via `struct.unpack_from` L176–L178); single-field path `_load_field` L188–L191 — https://github.com/modawan/reone/blob/master/src/libs/resource/format/gffreader.cpp#L68-L72

OffsetSizeIDTypeNote
0 ... entries FieldEntry Repeated `field_entry` (`GFFFieldData`); count `field_count`, base `field_offset`. Stride 12 bytes; consistent with `CResGFF::GetField` indexing (`0x00410990`).

Type: FieldData

Byte arena for complex field payloads; span = `field_data_count` from `field_data_offset` (`GFFHeaderInfo` +0x20 / +0x24).

OffsetSizeIDTypeNote
0 ... raw_data Opaque span sized by `GFFHeaderInfo.field_data_count` @ +0x24; base @ +0x20. Entries are addressed only through `GFFFieldData` complex-type offsets (not sequential). Per-type layouts: see `resolved_field` value_* instances and `bioware_common` types (CExoString, ResRef, LocString, vectors, binary). Wiki: https://github.com/OpenKotOR/PyKotor/wiki/GFF-File-Format#field-data

Type: FieldEntry

Type: FieldIndicesArray

Flat `u4` stream (`field_indices_count` elements from `field_indices_offset`). Multi-field structs slice this stream via `GFFStructData.data_or_data_offset`. “MultiMap” naming: PyKotor wiki (`wiki_gff_field_indices`) + Torlack ITP HTML (`xoreos_docs_torlack_itp_html`).

OffsetSizeIDTypeNote
0 ... indices u4le `field_indices_count` × `u4` from `field_indices_offset`. No per-row header on disk — `GFFStructData` for a multi-field struct points at the first `u4` of its slice; length = `field_count`. Ghidra: counts/offset from `GFFHeaderInfo` @ +0x28 / +0x2C.

Type: Gff3AfterAurora

GFF3 payload after the shared 8-byte Aurora prefix: `gff_header_tail` (48 B) then lazy arena instances.

Type: Gff4AfterAurora

GFF4 payload after the shared 8-byte Aurora prefix (through struct-template strip + remainder `tail`). PC-first LE numeric tail; `string_*` fields only when `aurora_version` (param) is V4.1.

OffsetSizeIDTypeNote
0 ... platform_id u4be Platform fourCC (`Header::read` first field). PC = `PC ` (little-endian payload); `PS3 ` / `X360` use big-endian numeric tail (not modeled byte-for-byte here).
4 ... file_type u4be GFF4 logical type fourCC (e.g. `G2DA` for GDA tables). `Header::read` uses `readUint32BE` on the endian-aware substream (`gff4file.cpp`).
8 ... type_version u4be Version of the logical `file_type` (GDA uses `V0.1` / `V0.2` per `gdafile.cpp`).
12 ... num_struct_templates u4le Struct template count (`readUint32` without BE — follows platform endianness; **PC LE** in typical DA assets). xoreos: `_header.structCount`.
16 ... string_count u4le V4.1 only — entry count for global shared string table (`gff4file.cpp` `Header::read`).
20 ... string_offset u4le V4.1 only — byte offset to UTF-8 shared strings (`loadStrings`).
24 ... data_offset u4le Byte offset to instantiated struct data (`GFF4Struct` root @ `_header.dataOffset`). `readUint32` on the endian substream (`gff4file.cpp`).
28 ... struct_templates Gff4StructTemplateHeader Contiguous template header array (`structTemplateStart + i * 16` in `loadStructs`).
??? ... tail Remaining bytes after the template strip (field-declaration tables at arbitrary offsets, optional V4.1 string heap, struct payload at `data_offset`, etc.). Parse with a full GFF4 graph walker or defer to engine code.

Type: Gff4File

Full GFF4 stream (8-byte Aurora prefix + `gff4_after_aurora`). Use from importers such as `GDA.ksy` that expect a single user-type over the whole file.

OffsetSizeIDTypeNote
0 ... aurora_magic u4be Aurora container magic (`GFF ` as `u4be`).
4 ... aurora_version u4be GFF4 `V4.0` / `V4.1` on-disk tags.
8 ... gff4 Gff4AfterAurora GFF4 header tail + struct templates + opaque remainder.

Type: Gff4StructTemplateHeader

OffsetSizeIDTypeNote
0 ... struct_label u4be Template label (fourCC style, read `readUint32BE` in `loadStructs`).

Type: GffHeaderTail

**GFF3** header continuation: **48 bytes** (twelve LE `u32` dwords) at file offsets **0x08–0x37**, immediately after the shared Aurora 8-byte prefix (`aurora_magic` / `aurora_version` on `gff_union_file`). Same layout as wiki [File Header](https://github.com/OpenKotOR/PyKotor/wiki/GFF-File-Format#file-header) rows from “Struct Array Offset” through “List Indices Count”. Ghidra `/K1/k1_win_gog_swkotor.exe`: `GFFHeaderInfo` @ +0x8 … +0x34.
